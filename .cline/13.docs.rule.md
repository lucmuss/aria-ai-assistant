# Doku-Regel: README als Hauptdokument (ausfuehrlich + SEO), restliche Docs kompakt

## Ziel
- `README.md` ist das zentrale, vollstaendige Produktdokument.
- Alle anderen Markdown-Dateien sind bewusst kompakt und verlinken auf die passenden README-Abschnitte.
- Die Doku bleibt fachlich korrekt, leicht verstaendlich und suchmaschinenfreundlich.

## 1) README-Standard (Pflicht)

### 1.1 Rolle
- `README.md` ist die **Single Source of Truth** fuer:
  - Produktuebersicht
  - Kern-Features
  - Installation/Setup
  - Konfiguration
  - Usage/Workflows
  - Troubleshooting
  - Release-/CI-Verhalten

### 1.2 Struktur (Mindestumfang)
- Klarer Titel (`# ...`) mit Produktname + Kernnutzen.
- Kurzbeschreibung direkt unter dem Titel (2-4 Saetze, klarer Nutzen).
- Inhaltsverzeichnis mit funktionierenden Anchor-Links.
- Abschnitte in dieser Reihenfolge:
  1. Was ist das Projekt?
  2. Features
  3. Installation
  4. Quickstart
  5. Konfiguration
  6. Nutzung (mit konkreten Beispielen)
  7. CI/CD und Release
  8. Sicherheit & Datenschutz
  9. Troubleshooting
  10. FAQ
  11. Beitragen / Entwicklung
  12. Lizenz

### 1.3 SEO- und Auffindbarkeitsregeln
- Wichtige Keywords natuerlich in Titel, Einleitung und 2-3 Ueberschriften platzieren.
- Erste 150-200 Zeichen muessen das Problem und den Nutzen klar benennen.
- Aussagekraeftige Ueberschriften mit suchbaren Begriffen (keine vagen Titel).
- Konsistente Terminologie (kein chaotischer Wechsel von Begriffen).
- Interne Links zwischen relevanten Abschnitten setzen.

### 1.4 Verstaendlichkeit
- Kurze Abschnitte, kurze Saetze, aktive Sprache.
- Keine unnötigen Buzzwords ohne konkrete Erklaerung.
- Copy-paste-faehige Kommandos und realistische Beispiele.
- Neueinsteiger sollen den ersten Erfolg in wenigen Minuten erreichen.

### 1.5 Aktualitaet
- Bei Feature-, Config-, Workflow- oder Release-Aenderungen muss README mitgezogen werden.
- Versionsangaben, Pfade, Dateinamen und Befehle muessen zum aktuellen Repo-Stand passen.

## 2) Andere Docs (kompakt)

### 2.1 Prinzip
- Weitere `.md` Dateien sind Kurz-Dokumente fuer Spezialthemen.
- Sie enthalten nur:
  - Zweck
  - Kernschritte
  - Stolpersteine
  - Link auf den passenden README-Abschnitt

### 2.2 Vermeidung von Duplikaten
- Keine vollstaendigen Wiederholungen aus README.
- Details gehoeren in README; Spezialdocs nur mit kompakten Ergänzungen.

## 3) Definition of Done fuer Doku-Aenderungen
- README ist fachlich korrekt und vollstaendig.
- SEO-Grundlagen sind erfuellt (klare Keywords, strukturierte Ueberschriften, gute Einleitung).
- Alle Kommandos sind copy-paste-faehig.
- Alle Links funktionieren.
- Spezialdocs sind kompakt und verweisen auf README statt Inhalte zu duplizieren.
